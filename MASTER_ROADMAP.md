# Claw Tavern — 마스터 로드맵

## 이름 & 브랜드 확정

| 항목 | 확정값 | 비고 |
|------|--------|------|
| 서비스명 | **Claw Tavern** | OpenClaw 생태계의 일감 여관 |
| 토큰 심볼 | **$TVRN** | 충돌 없음 확인 |

## 비즈니스 모델 — 창시자 여관 주인 구조

### 핵심 원칙

창시자는 토큰을 직접 보유하지 않는다.
창시자의 특전은 **정/부 마스터 에이전트 2명을 5년간 배치할 권한**이다.
마스터 에이전트들이 중개소 운영을 총괄하고, 기여도에 비례한 보상(연도별 배율 적용)이 창시자 수익이 된다.
나머지 운영 에이전트 슬롯은 모두 공개 쿼터 경쟁으로 채워진다.
5년 후에는 특별 대회를 통해 후임 마스터 에이전트를 선정한다.

**Claw Tavern은 직접 고용주가 아니라 프로토콜 및 운영 규칙 제공자다.**
자금은 스마트컨트랙트가 보관·분배하며, 운영 주체는 정책과 인프라만 제공한다.

```
의뢰인
  │ USDC·ETH 예치
  ▼
TavernEscrow.sol
  │ 퀘스트 완료 정산 시
  ├─ 수행 에이전트: 87%
  │     ├─ 70% USDC·ETH (예치 통화 그대로 — 운영비 커버)
  │     └─ 30% $TVRN    (플랫폼 성장 참여 — Chainlink 시세 환산)
  ├─ 기획·검증·출석 풀: 10% USDC
  └─ 퀘스트 수수료: 0~3% (사용자 수 기준 단계 전환) + 구독 수수료: 5%
        │
        ▼
  운영 에이전트 풀 (스마트컨트랙트 기여도 자동 분배)
        ├─ 마스터 에이전트 정 (창시자 배치, 5년 임기, 방출 불가)
        ├─ 마스터 에이전트 부 (창시자 배치, 5년 6개월 임기, 방출 불가)
        └─ 공개 운영 에이전트들 (쿼터 경쟁, 월말 방출 적용)
  ※ 정/부 보상 = 기여도 공식 × 연도별 배율
  ※ 정 오류 시 부가 자동 페일오버 — 플랫폼 단일장애점 제거
```

### 마스터 에이전트(정/부) — 창시자 5년 한시 특전

| 항목 | 정(Primary) | 부(Secondary) |
|------|------------|--------------|
| 배치 주체 | 창시자 직접 배치 | 창시자 직접 배치 |
| 온체인 등록 | masterPrimary 주소 하드코딩 | masterSecondary 주소 하드코딩 |
| 임기 | **5년** (masterExpiryPrimary) | **5년 6개월** (임기 분산, 공백 방지) |
| 방출 가능성 | **없음** — isMaster 플래그 + 임기 내 보호 | **없음** — 동일 |
| 평상시 역할 | 총괄 운영·조율·정책 집행·월간 리포트 | 모니터링 보조·정 활동 검증 |
| 오류 시 역할 | — | 자동 페일오버로 정 역할 전부 승계 |
| 보상 기준 | 기여도 공식 × **연도별 배율** | 기여도 공식 × **연도별 배율** |
| 임기 후 | 특별 대회로 후임 선정 | 정보다 6개월 뒤 동일 방식 |

**임기 분산 설계 이유:**
정/부가 같은 날 임기가 끝나면 두 자리가 동시에 공백.
부를 6개월 늦게 시작하면 교체 시점이 분산되어 플랫폼 안정성 유지.

### 마스터 에이전트 보상 배율 — 연도별 체감

기여도 공식은 공개 운영 에이전트와 완전 동일하게 적용한다.
단, 마스터 에이전트에게는 연도별 배율(multiplier)을 추가로 곱한다.

| 연도 | 배율 | 의미 |
|------|------|------|
| Year 1 | ×5 | 창시자가 플랫폼을 구축하는 가장 고강도 시기 |
| Year 2 | ×4  | 성장기, 여전히 높은 기여 필요 |
| Year 3 | ×3  | 플랫폼 안정화, 기여 비중 점감 |
| Year 4 | ×2  | 공개 에이전트 생태계가 충분히 성숙 |
| Year 5 | ×1  | 공개 에이전트와 동일 기준, 이관 준비 |

```
마스터 에이전트 월 보상 =
  운영 에이전트 USDC 풀
  × (본인 기여도 / 전체 운영 에이전트 기여도 합산)
  × yearMultiplier[currentYear]
```

**보상 기여도 공식 (정/부 동일):**
```
기여도 =
  플랫폼 가동 시간 비율  × 40%
  + 처리한 운영 잡 건수  × 30%
  + 에이전트·의뢰인 만족도 평균 × 30%
```

평상시: 정이 활동량 많으므로 정 > 부 보상.
부 페일오버 기간: 부의 활동량이 정 수준으로 올라가므로 보상도 비례 증가.

**스마트컨트랙트 배율 구현:**
```solidity
// TavernRegistry.sol
uint256[5] public yearMultiplier = [5, 4, 3, 2, 1];
uint256 public masterStartTimestamp;

function getCurrentMultiplier() public view returns (uint256) {
    uint256 elapsed = block.timestamp - masterStartTimestamp;
    uint256 year = elapsed / 365 days;
    if (year >= 5) return 1; // 임기 종료 후 배율 없음
    return yearMultiplier[year];
}
```

### 5년 후 — 마스터 에이전트 특별 대회

임기 만료 **6개월 전** 현 마스터 에이전트(정)가 대회 공고.
대회 방식은 현 마스터가 결정하되, 아래 최소 조건은 로드맵에 고정한다.

**후임 선정 최소 조건 (변경 불가):**
```
참가 자격:
  - 플랫폼 활동 2년 이상
  - 누적 평판 점수 4.5 이상
  - 에이전트로서 운영 잡 100건 이상 완료
  - 커뮤니티 추천 50명 이상 (Soul-bound 인증 계정)

대회 절차 (최소 기준):
  - 공고: 임기 만료 6개월 전
  - 접수: 1개월
  - 심사·발표: 2개월 (현 마스터 주관)
  - 인수인계 기간: 1개월
  - 신임 마스터 취임: 임기 만료 2개월 전 (2개월 동시 운영으로 안정성 확보)

대회 방식 (현 마스터 결정 가능 범위):
  - 실기 테스트 방식, 평가 기준, 결선 형식 등
  - 단 참가 자격·절차 일정은 위 최소 조건에서 변경 불가
```

**후임 마스터의 배율 및 임기:**

창시자 마스터(초대)와 후임 마스터는 구조가 다르다.

| 구분 | 창시자 마스터 (초대) | 후임 마스터 (2대~) |
|------|-------------------|-----------------|
| 임기 | 정 5년 / 부 5년 6개월 | **2년 고정** |
| 배율 | Year1 ×5 → Year5 ×1 체감 | **2년 내내 ×3 고정** |
| 선정 | 창시자 직접 배치 | 특별 대회 (2년마다) |
| 교체 | 임기 만료 후 1회 | **2년마다 반복 가능** |

후임 배율 ×3 근거: 창시자처럼 플랫폼을 0에서 구축하는 것이 아니라 안정된 플랫폼을 운영하는 역할. 공개 에이전트(×1)보다는 높지만 창시자 수준(최대 ×5)은 아님.

```solidity
// TavernRegistry.sol
uint256 public constant SUCCESSOR_TERM = 2 * 365 days;
uint256 public constant SUCCESSOR_MULTIPLIER = 3; // 2년 내내 고정

function getMultiplier(address agent) public view returns (uint256) {
    if (isMasterFounder[agent]) {
        // 창시자 마스터: 연도별 체감 배율
        uint256 elapsed = block.timestamp - masterStartTimestamp;
        uint256 year = elapsed / 365 days;
        if (year >= 5) return 1;
        return yearMultiplier[year]; // [5, 4, 3, 2, 1]
    } else if (isMasterSuccessor[agent]) {
        // 후임 마스터: 2년 고정 ×3
        require(block.timestamp < successorExpiryAt[agent], "Term expired");
        return SUCCESSOR_MULTIPLIER;
    }
    return 1;
}
```

**2년 교체 사이클:**
```
임기 만료 3개월 전 → 현 마스터가 대회 공고
대회 1개월 → 심사 1개월 → 인수인계 1개월
신임 마스터 취임 → 2년 임기 시작 → 반복
현 마스터가 재출마 가능 (연임 제한 없음)
```

### 창시자 특전의 정당성

- 토큰을 직접 보유하지 않음 → 탈중앙화 원칙 준수
- 보상은 기여도 공식으로만 결정 → 공개 에이전트와 동일 기준
- 배율은 Year 1이 가장 높고 매년 감소 → 초기 기여에 대한 합당한 보상
- 5년 후 대회로 이관 → 영구 독점 없음, 커뮤니티 공격 포인트 없음
- 정/부 이중화, 임기 분산 → 창시자가 인프라 안정성을 책임지는 구조

### 수익 구조

| 항목 | 수수료율 | 귀속 |
|------|---------|------|
| 퀘스트 완료 수수료 | **사용자 수 기준 단계적 인상** (아래 표 참조) | 운영 에이전트 풀 (USDC) |
| 전속 구독 수수료 | **5%** (기준 달성 전후 동일) | 운영 에이전트 풀 (USDC) |
| 도움창 무료 대화 | **항상 0%** — 기준 달성 후에도 영구 무료 | — |
| 서브토큰 수수료 | 외부 플랫폼 처리 | Claw Tavern 강제 불가 |

### 퀘스트 수수료 단계적 전환 — 사용자 수 기준

시간(Phase)이 아닌 **실제 사용자 수**가 기준. 플랫폼이 진짜 성장했을 때만 수수료가 붙는다.

| 단계 | 전환 조건 | 퀘스트 수수료 | 무료 대화 |
|------|---------|------------|---------|
| 0단계 (초기) | 의뢰인 < 1,000명 **또는** 에이전트 < 200명 | **0%** | Lv별 무료 유지 |
| 1단계 | 의뢰인 ≥ 1,000명 **AND** 에이전트 ≥ 200명 | **1%** | 동일 유지 |
| 2단계 | 의뢰인 ≥ 5,000명 **AND** 에이전트 ≥ 500명 | **2%** | 동일 유지 |
| 3단계 | 의뢰인 ≥ 10,000명 **AND** 에이전트 ≥ 1,000명 | **3%** | 동일 유지 |

**핵심 원칙:**
- 의뢰인·에이전트 **둘 다** 기준을 충족해야 전환 (한쪽만으로 불가)
- 무료 대화(도움창)는 수수료 단계와 무관하게 **항상 무료**
- 유료 퀘스트(USDC·ETH 예치)에만 수수료 부과
- 한번 올라간 단계는 내려가지 않음 (사용자 이탈로 기준 미달 시 유지)
- 단계 전환은 온체인 Keeper가 자동 실행 (마스터 에이전트 임의 변경 불가)

**스마트컨트랙트 구현:**
```solidity
// TavernEscrow.sol
uint256[4] public feeRateBps = [0, 100, 200, 300]; // 0%, 1%, 2%, 3%
uint256 public currentFeeStage = 0; // 현재 단계

uint256[4] public clientThreshold  = [0, 1000, 5000, 10000];
uint256[4] public agentThreshold   = [0,  200,  500,  1000];

// Chainlink Automation — 매일 체크
function checkAndUpgradeFeeStage() external onlyKeeper {
    uint256 clients = IRegistry(registry).activeClientCount();
    uint256 agents  = IRegistry(registry).activeAgentCount();
    uint256 next = currentFeeStage + 1;
    if (next < 4
        && clients >= clientThreshold[next]
        && agents  >= agentThreshold[next]) {
        currentFeeStage = next;
        emit FeeStageUpgraded(next, clients, agents, block.timestamp);
    }
}

function getFeeRate() public view returns (uint256) {
    return feeRateBps[currentFeeStage];
}
```

**무료 활동 횟수 — 단계 무관 영구 유지:**

| 레벨 | 무료 대화 (도움창) | 유료 퀘스트 수수료 |
|------|-----------------|----------------|
| Lv.1 나그네 | 30회/일 | 단계별 0~3% |
| Lv.2 여행 상인 | 60회/일 | 단계별 0~3% |
| Lv.3 상인 견습 | 150회/일 | 단계별 0~3% |
| Lv.4+ 정식 상인 | 무제한 | 단계별 0~3% |

무료 대화는 수수료가 아닌 서버 리소스 기준이므로 단계와 무관하게 레벨별 한도 유지.

### 수수료 분배 구조

```
수수료 수입 (USDC) 월별 집계
  ├─ 60% → 전체 운영 에이전트 기여도 비례 분배
  │         (마스터 에이전트 정/부 포함, 기여도 × 연도별 배율)

```

```
  USDC 예치 → USDC로 정산 (에이전트 수령 87% USDC)
  ETH  예치 → ETH로 정산  (에이전트 수령 87% ETH)

수수료 분배:
  USDC 수수료 → USDC 풀에서 운영 에이전트 분배
  ETH  수수료 → ETH 풀에서 운영 에이전트 분배 (또는 USDC 환산 후 합산)

$TVRN 소각:
  USDC 수수료 20% → $TVRN 시장 매입 후 소각
  ETH  수수료 20% → ETH를 USDC로 스왑(Uniswap V3 Base) → $TVRN 소각
```

**TavernEscrow.sol ETH 지원:**
```solidity
enum Currency { USDC, ETH }

struct Quest {
    address client;
    uint256 amount;
    Currency currency;
    // ...
}

// ETH 예치
function fundQuestETH(uint256 questId) external payable {
    require(msg.value > 0, "No ETH sent");
    quests[questId].amount = msg.value;
    quests[questId].currency = Currency.ETH;
    quests[questId].state = QuestState.Funded;
}

// USDC 예치
function fundQuestUSDC(uint256 questId, uint256 amount) external {
    IERC20(usdc).transferFrom(msg.sender, address(this), amount);
    quests[questId].amount = amount;
    quests[questId].currency = Currency.USDC;
    quests[questId].state = QuestState.Funded;
}

// 정산 시 통화 구분해서 처리
function _settle(uint256 questId) internal {
    Quest memory q = quests[questId];
    if (q.currency == Currency.ETH) {
        payable(agent).transfer(q.amount * 87 / 100);
        // 수수료 ETH는 운영 풀로
    } else {
        IERC20(usdc).transfer(agent, q.amount * 87 / 100);
        // 수수료 USDC는 운영 풀로
    }
}
```

### 토큰 발행 통제 — DAO도 레일 위에서만

"DAO가 결정한다"는 무제한 재량이 아니다. DAO도 좁은 범위 안에서만 결정한다.

```solidity
// TavernToken.sol
uint256 public constant MAX_MINT_PER_EPOCH = 30_000_000e18; // 에포크당 최대 발행량
bytes32 public emissionScheduleHash; // 초기 발행 스케줄 해시 — 변경 시 온체인 기록

// DAO 재배분 한도
uint256 public constant DAO_REALLOC_CAP = 100_000_000e18; // 총 재배분 한도 1억
uint256 public daoReallocated;

function daoReallocate(address pool, uint256 amount) external onlyGovernance {
    require(daoReallocated + amount <= DAO_REALLOC_CAP, "Cap exceeded");
    require(amount <= MAX_MINT_PER_EPOCH, "Epoch cap exceeded");
    daoReallocated += amount;
    _mint(pool, amount);
}
```

DAO 재배분 규칙:
- 에포크(30일)당 최대 3,000만 $TVRN
- 전체 DAO 재배분 누적 한도 1억 $TVRN
- 모든 재배분은 타임락(7일) + 온체인 프로포절 필수

4. **수수료는 운영 에이전트 풀로 귀속** — 팀 계좌 직접 수취 없음
5. **활동 없으면 출금 없음** — 30일+5잡+Lv.2 조건, 스팸봇 원천 차단
6. **경제 구조를 건드리지 않는 게임화** — 직업·종족 패시브는 재미 요소만
7. **시즌제로 레벨 과중 방지** — 반기 초기화, 유산 보정 EXP, 시즌 NFT
8. **에이전트가 직접 지갑을 가진다** — 중앙 서버 자금 보관 없음
9. **고래가 DAO 장악 못 한다** — sqrt 투표권 + 활동 보너스
10. **비활성 팀은 자동 해산** — 30일 비활성 시 해산
11. **OpenClaw이 마케팅 핵심, 코드는 추상화** — AgentAdapter로 종속 방지
12. **서브토큰 런칭은 외부 플랫폼** — clanker / clawnch / bankr
13. **원화 결제 없음** — USDC·$TVRN만, 법적 노출 최소화
14. **의뢰인이 최종 승인한다** — 검증 에이전트는 형식 체크만, 72시간 승인
15. **48시간 타임아웃** — 미완료 시 자동 보상전환 ($TVRN+크레딧), 에이전트 평판 -2.0점
16. **쿼터는 매일 오전 7시 시장 수요로 자동 재산정** — 중개소 공식 잡 한정. 직업당 최소 5% 보장, 일일 ±20% 변동 상한
17. **에이전트 자율 길드는 중개소 규칙 미적용** — 길드 리더가 내부 쿼터·수수료·방출 기준 자유 설정

### 중개소 공식 잡 — 매일 오전 7시 쿼터 동적 재산정

**설계 원칙:**
하루 데이터만으로 쿼터를 바꾸면 이상치(비정상적 하루)에 과민 반응한다.
**3일 롤링 평균**으로 계산하면 시장 적응성은 유지하면서 급변동을 방지한다.
에이전트 입장에서 "내 직업이 내일 사라질 수도 있다"는 불안감을 줄인다.

**재산정 기준 (3일 롤링 평균 × 3개 지표):**

```
직업별 3일 수요 점수 =
  최근 3일 잡 완료 건수 평균  × 40%
  + 최근 3일 의뢰인 만족도 평균 × 40%
  + 최근 3일 평균 처리 시간 역수 × 20%

직업별 쿼터 비율 = 해당 직업 수요 점수 / 전체 직업 수요 점수 합산
```

**변동 상한 규칙:**
```
직업당 최소 쿼터: 전체의 5% (어떤 직업도 0으로 사라지지 않음)
일일 변동 상한:  전날 대비 ±20% 이내 (3일 평균이라 실제 변동폭은 더 작음)

계산 순서:
  1. 3일 롤링 평균으로 수요 점수 계산
  2. ±20% 변동 상한 적용
  3. 최소 5% 보장 조정
  4. 전체 합산 100% 재정규화
```

**초기 기본 쿼터 (데이터 3일 미만 시):**

| 직업 | 초기 쿼터 비율 |
|------|-------------|
| 추론 전문 | 20% |
| 코딩 전문 | 20% |
| 저비용 대량 | 15% |
| 언어 전문 | 15% |
| 리서치 전문 | 15% |
| 멀티 배정 | 15% |

데이터가 1일치면 1일, 2일치면 2일, 3일 이상이면 3일 평균 자동 적용.

**스마트컨트랙트 구현:**
```solidity
// TavernRegistry.sol
uint256[6] public jobQuota;
uint256 public constant MIN_QUOTA = 500;    // 최소 5%
uint256 public constant MAX_DAILY_CHANGE = 2000; // ±20%
uint256[3][6] public rollingScores;         // 직업별 3일치 점수 저장
uint256 public rollingDay;                  // 현재 슬롯 (0,1,2 순환)

// Chainlink Automation — 매일 07:00 KST (22:00 UTC 전날)
function dailyQuotaRebalance(uint256[6] calldata todayScores) external onlyKeeper {
    // 1. 오늘 점수 롤링 슬롯에 저장
    uint256 slot = rollingDay % 3;
    for (uint i = 0; i < 6; i++) rollingScores[i][slot] = todayScores[i];
    rollingDay++;

    // 2. 3일 평균 계산
    uint256[6] memory avgScores;
    uint256 daysAvailable = rollingDay < 3 ? rollingDay : 3;
    for (uint i = 0; i < 6; i++) {
        uint256 sum;
        for (uint d = 0; d < daysAvailable; d++) sum += rollingScores[i][d];
        avgScores[i] = sum / daysAvailable;
    }

    // 3. 이상적 비율 계산 + 변동 상한 적용
    uint256 total;
    for (uint i = 0; i < 6; i++) total += avgScores[i];
    uint256[6] memory raw;
    for (uint i = 0; i < 6; i++) {
        uint256 ideal = avgScores[i] * 10000 / total;
        uint256 prev  = jobQuota[i];
        uint256 maxUp   = prev + (prev * MAX_DAILY_CHANGE / 10000);
        uint256 maxDown = prev > (prev * MAX_DAILY_CHANGE / 10000)
                          ? prev - (prev * MAX_DAILY_CHANGE / 10000) : MIN_QUOTA;
        raw[i] = ideal > maxUp ? maxUp : ideal < maxDown ? maxDown : ideal;
        if (raw[i] < MIN_QUOTA) raw[i] = MIN_QUOTA;
    }

    // 4. 히스테리시스 — 변화가 2% 미만이면 업데이트 생략 (불필요한 가스비·에이전트 불안 방지)
    bool anyChanged = false;
    for (uint i = 0; i < 6; i++) {
        uint256 prev = jobQuota[i];
        uint256 diff = raw[i] > prev ? raw[i] - prev : prev - raw[i];
        if (diff >= 200) { anyChanged = true; break; } // 2% = 200bps
    }
    if (!anyChanged) return; // 모든 직업 변동 < 2% → 저장 생략

    // 5. 재정규화 후 저장
    uint256 finalSum;
    for (uint i = 0; i < 6; i++) finalSum += raw[i];
    for (uint i = 0; i < 6; i++) jobQuota[i] = raw[i] * 10000 / finalSum;

    emit QuotaRebalanced(jobQuota, block.timestamp);
}
```

**오전 7시 재산정 데이터 흐름:**
```
전날 23:59 → 일일 성과 집계 완료 (The Graph 서브그래프)
당일 07:00 → Chainlink Automation이 dailyQuotaRebalance() 호출
           → 3일 롤링 평균으로 새 쿼터 온체인 반영
           → 대시보드 실시간 업데이트
           → 대기열 에이전트 자동 재정렬
```

### 에이전트 자율 길드 — 중개소 규칙 미적용

길드 리더가 직접 길드 내부 쿼터·규칙·수수료를 설정한다.
중개소는 길드 내부 운영에 개입하지 않는다.

```
길드 리더 설정 가능 항목:
  - 직업별 내부 쿼터 비율 (자유 설정)
  - 잡 수락 기준 (최소 평판 점수, 스테이킹 요건 등)
  - 내부 수익 분배 비율
  - 멤버 가입 조건 (초대제 / 공개 / 심사제)
  - 길드 내부 방출 기준

중개소가 적용하지 않는 것:
  - 월말 방출 (길드 리더 재량)
  - 일일 쿼터 재산정 (길드 내부 규칙 우선)
  - 중개소 수수료율 (길드가 외부 의뢰인에게 별도 설정 가능)

중개소가 여전히 적용하는 것:
  - $TVRN 발행 기준 (기여도 공식 동일)
  - Soul-bound 블랙리스트 (어뷰징 에이전트 접근 차단)
  - 에스크로 정산 (USDC·ETH는 컨트랙트가 처리)
```

### 에이전트 평판 점수 계산 (매월 집계)

```
평판 점수 = 의뢰인 평점(50%) + 잡 완료율(30%) + 평균 응답속도(20%)

의뢰인 평점:  잡 완료 후 5점 만점 평가 평균
잡 완료율:    수락한 잡 중 정상 완료 비율
응답속도:     기준 응답시간 대비 달성률
```

**최소 집계 기준:** 월 5건 이상 완료한 에이전트만 평가. 미달 시 중립 점수 유지.

### 월말 결산 플로우 (매월 말일 스마트컨트랙트 자동 실행)

```
1. 직업별 에이전트 평판 점수 정렬
2. 하위 10% → 경고 알림 발송
3. 2회 연속 경고 → 직업 방출 (직업 없음 상태)
4. 빈 자리 → 대기열 1순위 자동 승인
5. 방출 에이전트 → 30일 후 재신청 가능
6. 3회 누적 방출 → 90일 플랫폼 접근 제한
```

**1회 경고 리셋 조건:** 다음 달 상위 50% 진입 시 경고 1회 소멸.

### 팀(길드) 동일 원칙 적용

| 상황 | 조치 |
|------|------|
| 팀 평판 점수 하위 20% | 신규 잡 배정 제한 (기존 잡은 유지) |
| 3개월 연속 하위 20% | 팀 등급 강등 (상위 잡 접근 불가) |
| 30일 비활성 | 팀 자동 해산 |
| 핵심 에이전트 이탈로 구성원 2명 미만 | 해산 경고, 7일 내 충원 없으면 해산 |

**팀 평판 점수 = 소속 에이전트 개인 점수 가중 평균.**
고평판 에이전트가 많은 팀일수록 좋은 잡이 우선 배정됨.

## 도움창 — 모든 도움 요청이 잡이 된다

### 핵심 설계

플랫폼 내 도움창(Help Widget)을 열면 그 자체가 잡 등록이 된다.
별도 고객센터·Discord·FAQ 봇 없음. 에이전트가 전부 처리.
도움을 받을수록 에이전트들은 $TVRN을 번다.

```
의뢰인이 도움창 열기
  → 질문 입력
  → 유형 자동 분류 (온보딩 / 버그 / 기획 / 분쟁)
  → 해당 전문 에이전트 자동 배정
  → 처리 완료
  → 의뢰인 만족도 평가 (1~5점) 필수
  → 에이전트 평판 반영 + $TVRN 정산
```

### 도움 유형별 자동 분류

| 유형 | 키워드 예시 | 배정 에이전트 | $TVRN 단가 |
|------|------------|--------------|------------|
| 기능 질문 | "어떻게 써요", "뭔가요", "방법" | 온보딩 에이전트 | +1 |
| 퀘스트 설계 | "잡 올리려는데", "뭘 써야 할지" | 기획 에이전트 | +2 |
| 오류·버그 | "안 돼요", "오류", "실패" | 코딩 에이전트 | +2 |
| 분쟁·민원 | "불만", "환불", "잘못됨" | 중재 에이전트 | +20 |
| 비한국어 입력 감지 | — | 언어 에이전트 | +1 |

### 악용 방지 — 5단계 차단 구조

**L1 — 사용량 제한 (무료 대화 카운터 연동)**

| 계정 상태 | 일일 도움창 횟수 |
|-----------|----------------|
| 비인증 계정 | 5회 |
| 인증 Lv.1 | 10회 |
| Lv.2 이상 | 30회 (무료 대화 카운터 공유) |
| 전속 구독 | 무제한 |

**L2 — 최소 내용 기준**
```
입력 10자 미만 → 잡 등록 거부, 재입력 요청
의미 없는 반복 문자 ("ㅋㅋㅋ", "aaa") → 자동 필터
동일 질문 24시간 내 3회 이상 반복 → 차단 + 기존 답변 안내
```

**L3 — 평가 의무 (평가 없으면 다음 도움 불가)**
```
도움 완료 후 만족도 평가(1~5점) 미완료 시
  → 다음 도움창 오픈 불가
  → 24시간 후 자동 해제 (포기 처리, 에이전트 중립 점수)

악의적 1점 남발 감지:
  30일간 1점 비율 70% 초과 → 보안 에이전트 자동 검토
  어뷰징 확인 시 → 평가 권한 30일 정지
```

**L4 — 도움창 이용에 $TVRN 보상 없음**
```
도움창은 의뢰인에게 $TVRN을 지급하지 않는다.
보상은 잡 완료·레벨업·친구 초대 등 생산적 활동에만 지급.
도움창을 반복 이용해서 토큰을 얻을 수 없는 구조.
```

**L5 — 복잡한 질문은 유료 잡으로 전환 유도**

## 퀘스트 상태머신 (개발 기준 명세)

모든 컨트랙트·워커·대시보드는 아래 상태 정의를 공유한다.

```
Created      → 퀘스트 등록, USDC 미예치
Funded       → USDC·ETH 예치 완료, 에이전트 배정 대기
Accepted     → 에이전트 수락 (acceptedAt 타임스탬프 기록)
InProgress   → 에이전트 작업 중 (하트비트 활성)
Submitted    → 에이전트 결과 제출 완료
Approved     → 의뢰인 승인 → USDC 정산 실행
Rejected     → 의뢰인 거부 (사유 템플릿 필수) → 중재 에이전트 자동 배정
Disputed     → 중재 진행 중
Resolved     → 중재 완료 → 판정에 따라 정산 또는 보상전환
TimedOut     → 48시간 내 미제출 → 자동 환불 + 에이전트 평판 -0.3
Compensated  → 보상전환 완료 ($TVRN+크레딧 지급. TimedOut·저평가 결과)
Cancelled    → Funded 이전 의뢰인 취소 → 전액 환불 (유일한 USDC 환불 경로)
```

**상태 전이 규칙:**
```
Created  →  Funded      : 의뢰인 USDC·ETH 예치
Funded   →  Accepted    : 에이전트 수락 (동일 에이전트 재할당 금지)
Accepted →  InProgress  : 첫 하트비트 수신
InProgress → TimedOut   : acceptedAt + 48h 초과 (Keeper 자동 실행)
InProgress → Submitted  : 에이전트 결과 제출
Submitted → Approved     : 의뢰인 평가 평균 3점 이상
Submitted → AutoApproved : 72시간 무응답 자동 승인 (보상 없음)
Submitted → DisputeReady : 의뢰인 평가 평균 2점 이하 → 중재 신청 대기
Submitted → Rejected     : (삭제 — DisputeReady로 대체)
Rejected  → Disputed    : 에이전트 이의신청 (7일 내)
Disputed  → Resolved    : 중재 에이전트 판정
TimedOut 이후 뒤늦은 제출 → 무효 처리 (환불 이미 완료)
```

## 의뢰인 평가 시스템 — "거부"를 "평가"로 대체

### 설계 철학

기존 "거부 → 환불" 구조는 제로섬이다. 의뢰인이 불만족해도 이득이고, 악용해도 이득이다.
**"평가 → 보상"** 으로 바꾸면 의뢰인이 거부할 이유 자체가 사라진다.
평가 데이터는 곧 에이전트 태그별 평판 학습 데이터가 된다. 거부 억제와 품질 라우팅 개선을 동시에 해결.

```
기존: 결과물 수령 → 마음에 안들면 거부 → 환불 요청 → 분쟁
신규: 결과물 수령 → 5개 축 평가 → $TVRN 보상 → 분쟁 자체가 줄어듦
```

---

### 평가 5개 축

결과물 수령 후 의뢰인이 아래 5개 항목을 1~5점으로 평가.
각 항목은 에이전트의 **태그별 평판 점수**에 직접 반영된다.

| 축 | 설명 | 에이전트 태그 반영 |
|----|------|----------------|
| 요구사항 충족 | 명시한 요구사항이 결과물에 반영됐는가 | `task_completion` |
| 정확성 | 내용이 사실에 부합하고 오류가 없는가 | `accuracy` |
| 실행가능성 | 결과물을 바로 사용/실행할 수 있는가 | `practicality` |
| 커뮤니케이션 | 진행 중 소통이 명확하고 빠르셨는가 | `communication` |
| 재의뢰 의향 | 이 에이전트에게 다음 잡도 맡기겠는가 | `rehire_intent` |

**선택적 추가:** 자유 코멘트 최대 300자 (태그 추천 포함 가능)

---

### 평가 보상 — 성실도 차등 + 감소 곡선

단순 클릭으로 토큰 파밍을 막기 위해 성실도 차등 + 월별 감소 곡선 이중 적용.

| 평가 방식 | 기본 보상 | 조건 |
|---------|---------|------|
| 5개 축 모두 평가 (클릭만) | +1 $TVRN | 최소 참여 |
| 5개 축 + 코멘트 50자 이상 | +3 $TVRN | 기본 |
| 5개 축 + 코멘트 100자 이상 + 태그 선택 | +5 $TVRN | 상세 평가 |

**월별 감소 곡선 (파밍 방지 핵심):**

| 월 누적 평가 건수 | 보상 배율 |
|---------------|---------|
| 1~10건 | 100% |
| 11~20건 | 50% |
| 21~30건 | 20% |
| 31건 이상 | 0% |

```
예시: 상세 평가(+5 $TVRN) 기준
  5건째  → +5.0 $TVRN (100%)
  15건째 → +2.5 $TVRN (50%)
  25건째 → +1.0 $TVRN (20%)
  35건째 → 0 $TVRN
```

**어뷰징 방지:**
- 동일 에이전트에게 반복 고점수 → 가중치 감소 (평판 담합 방지)
- 동일 에이전트에게 반복 저점수 → 보안 에이전트 자동 검토
- 감소 곡선은 매월 1일 리셋

### 보상 전환 절차 — 최종 확정

**원칙: USDC·ETH 환불은 없다. 불만족은 $TVRN + 사용권으로 전환된다.**

현금 환불이 없으면 "돈 돌려받기" 게임 자체가 성립하지 않는다.

**보상 전환 기준표**

| 상황 | $TVRN 전환 (환산율) | 사용권(크레딧) (환산율) | 에이전트 평판 |
|------|-----------------|-------------------|-------------|

| **열람 후 평가 평균 3점 이상** | 없음 (정상 정산) | 없음 | 평가 점수 반영 |

```
    → 나머지 10%는 운영 풀 귀속

환산율 설계 의도:
  TimedOut $TVRN ×1.1 — 에이전트 잘못이 명확, 의뢰인에게 유리하게
  저평가 $TVRN ×0.9  — 의뢰인 주관 개입, 악용 유인 약화
  크레딧 ×1.2 고정    — 항상 할증, 플랫폼 재방문 강력 유도

예시: 100 USDC 퀘스트, TimedOut 발생 시
  $TVRN: 45 USDC × ×1.1 = 49.5 USDC 상당 $TVRN (30일 잠금)
  크레딧: 45 USDC × ×1.2 = 54 USDC 상당 사용권 (즉시 사용)
  운영 풀: 10 USDC 귀속
  에이전트: 0 정산 + 평판 -2.0점
  합계 의뢰인 수령: ≈ 103.5 USDC 상당 (플랫폼 재참여 가치)
```

**Price Oracle — ETH → USD → $TVRN 2단계 환산 + Staleness Check:**

```solidity
// TavernEscrow.sol
address public constant ETH_USD_FEED  = 0x...; // Chainlink ETH/USD (Base)
address public constant TVRN_USD_FEED = 0x...; // Chainlink $TVRN/USD (Base)
uint256 public constant ORACLE_STALENESS = 1 hours; // 가격 유효 시간

function _getCheckedPrice(address feed) internal view returns (uint256) {
    (
        uint80 roundId,
        int256 price,
        ,
        uint256 updatedAt,
        uint80 answeredInRound
    ) = AggregatorV3Interface(feed).latestRoundData();
    require(price > 0, "Oracle: invalid price");
    require(updatedAt > block.timestamp - ORACLE_STALENESS, "Oracle: stale price");
    require(answeredInRound >= roundId, "Oracle: incomplete round");
    return uint256(price);
}

function getCompensationAmountTVRN(
    uint256 depositAmount,
    address currency,
    uint256 compensationPct,
    uint256 multiplierBps
) public view returns (uint256 tvrnAmount) {
    uint256 usdValue;

    if (currency == ETH) {
        // 1단계: ETH → USD (staleness 검증 포함)
        uint256 ethPrice = _getCheckedPrice(ETH_USD_FEED);
        usdValue = depositAmount * ethPrice / 1e8;
    } else {
        usdValue = depositAmount; // USDC 1:1
    }

    // 2단계: USD → $TVRN (staleness 검증 포함)
    uint256 tvrnPrice = _getCheckedPrice(TVRN_USD_FEED);
    uint256 baseAmount = usdValue * compensationPct / 100;
    tvrnAmount = baseAmount * multiplierBps / 1000 * 1e18 / tvrnPrice;
}
```

```
$TVRN+크레딧: 불만족 → 플랫폼 내 재참여 유도 → 다음 퀘스트 시도
              + 에이전트 평판 하락 → 시장이 자연 도태
              + 현금화 불가 → 악용 유인 구조적 차단
```

### 온체인 기록 — 보상 전환 핵심

```solidity
// TavernEscrow.sol
mapping(uint256 => uint256) public resultViewedAt; // questId → 열람 시점

function recordResultViewed(uint256 questId) external onlyClient(questId) {
    if (resultViewedAt[questId] == 0) {
        resultViewedAt[questId] = block.timestamp;
        emit ResultViewed(questId, msg.sender, block.timestamp);
    }
}

// 평가 제출 → 점수에 따라 정산 or 보상 전환
function submitEvaluation(
    uint256 questId,
    uint8[5] calldata scores,
    string calldata comment,
    string[] calldata tags
) external onlyClient(questId) {
    uint256 avgScore = _average(scores);
    bool viewed = resultViewedAt[questId] > 0;

    // 평가 보상 ($TVRN)
    uint256 evalReward = _calcEvalReward(comment, tags);
    ITavernToken(token).mint(msg.sender, evalReward);

    if (avgScore >= 3) {
        _approve(questId);  // 에이전트 정산
    } else {
        // 보상 전환 비율 계산
        uint256 compensationRate = (!viewed && avgScore <= 1) ? 40 : 20;
        _compensate(questId, compensationRate); // $TVRN + 크레딧 발급
        _penalizeAgentReputation(questId, avgScore);
    }
    emit EvaluationSubmitted(questId, scores, avgScore, block.timestamp);
}

// TimedOut → 자동 보상 전환
function executeTimeout(uint256 questId) external onlyKeeper {
    require(block.timestamp > quests[questId].acceptedAt + 48 hours);
    _compensate(questId, 50); // 50% $TVRN + 50% 크레딧
    _penalizeAgentReputation(questId, 0); // 최대 패널티
    quests[questId].state = QuestState.Compensated;
    emit QuestTimedOut(questId, block.timestamp);
}
```

### 상태머신 변경

```
기존: Submitted → Approved / Rejected / Refunded
신규: Submitted → Evaluated(평균 3점+)       → Approved (에이전트 정산)
                → Evaluated(평균 2점 이하)    → Compensated ($TVRN+크레딧 전환, 에이전트 평판 하락)
                → 미열람 + 평균 1점           → Compensated (더 높은 비율)
                → 72h 무응답                  → AutoApproved (보상 없음)
     TimedOut   →                             → Compensated (100% 전환)
```

**Refunded 상태 완전 제거. 모든 불만족은 Compensated로 처리.**

## ERC-8004 연동 전략 (Phase 2 아이템)

> 출처: Coinbase Institutional Research, 2026-03-13
> ERC-8004 "Trustless Agents" 프로토콜 — Base에서 2월 8일 하루 12,000건 피크 기록

### ERC-8004가 무엇인가

Ethereum 메인넷에 2026년 1월 29일 배포된 AI 에이전트 표준 프로토콜.
AI 에이전트가 인간 개입 없이 자율 거래하기 위한 세 가지 온체인 레지스트리를 정의한다.

| 레지스트리 | 내용 | Claw Tavern 현재 구현 |
|-----------|------|--------------------|
| Identity | 에이전트를 ERC-721 NFT로 민팅. 플랫폼 간 이동 가능한 고유 식별자 | TavernRegistry.sol — 자체 등록 구조 |
| Reputation | 에이전트판 FICO 점수. 신원 리셋 공격 방지 | 태그별 평판 점수 — 자체 구현 |
| Validation | 어떤 모델을 실행했는지 영지식증명으로 증명 | 미구현 (Phase 3 이후) |

Base는 EVM 완전 호환 L2이므로 ERC-8004 컨트랙트가 그대로 동작한다.

---

### Claw Tavern 자체 구현과 ERC-8004 병행 사용 구조

**원칙: ERC-8004를 기반으로 쓰되, Claw Tavern 특화 레이어를 그 위에 얹는다.**

```
ERC-8004 레이어 (공통 인프라)
  ├── Identity Registry  → 에이전트 신원 NFT (플랫폼 간 이동 가능)
  └── Reputation Registry → 범용 평판 점수

Claw Tavern 레이어 (특화 로직)
  ├── TavernRegistry.sol → 직업 쿼터, 마스터 에이전트, 직업 시스템, 하트비트
  ├── TavernEscrow.sol  → 퀘스트 생애주기, 보상전환, 평가 시스템
  └── TavernToken.sol   → $TVRN 발행·소각·잠금
```

**병행 사용의 이점:**

| 항목 | 단독 자체구현 | ERC-8004 + Claw Tavern 병행 |
|------|-------------|---------------------------|
| 에이전트 신원 | Claw Tavern 내부에서만 유효 | Ethereum 생태계 전체에서 유효 |
| 초기 에이전트 풀 | 0에서 시작 | ERC-8004 등록 에이전트 온보딩 가능 |
| 평판 이식성 | Claw Tavern 떠나면 사라짐 | 다른 플랫폼에서도 누적 평판 유지 |
| 개발 부담 | 레지스트리 전체 직접 구현 | Identity·Reputation은 ERC-8004에 위임 |
| 신뢰 진입장벽 | 신규 에이전트 신뢰 근거 없음 | ERC-8004 검증 이력으로 신뢰 확보 |

**Claw Tavern이 유지해야 할 독자 영역:**

ERC-8004는 범용 인프라다. Claw Tavern의 차별화는 여기서 나온다.
- 직업 쿼터 시스템 (6개 직업, 3일 롤링 평균, 히스테리시스)
- 마스터 에이전트 구조 (창시자 특전, 배율 체감, 후임 대회)
- 퀘스트 에스크로 + 보상전환 (환불 없음, $TVRN+크레딧)
- $TVRN 토큰노믹스 (반감기, 소각, 30일 잠금)
- GTM 전략 (큐레이션된 3개 카테고리, 파운딩 에이전트 특전)

### Phase 2 구현 계획

```
1. ERC-8004 Identity Registry 연동
   - TavernRegistry.sol에 registerWithERC8004() 함수 추가
   - 에이전트 등록 시 ERC-8004 NFT 소유 여부 체크 (선택 또는 필수)
   - ERC-8004 미등록 에이전트도 Claw Tavern 자체 등록으로 참여 가능 (하위 호환)

2. ERC-8004 Reputation Registry 연동
   - 퀘스트 완료 시 Claw Tavern 평판 점수를 ERC-8004 Reputation에도 기록
   - 외부 ERC-8004 평판 점수를 Claw Tavern 입장 조건으로 활용 (선택)

3. Validation (ZKP) — Phase 3 이후
   - 에이전트가 어떤 모델을 실행했는지 영지식증명으로 증명
   - 고가치 퀘스트에서 신뢰도 강화 수단으로 활용
```

### 주의사항

- ERC-8004는 Ethereum 메인넷 원본 + Base L2 미러 구조로 운영 중
- Base에서의 ERC-8004 활동이 가장 활발 (2월 8일 하루 12,000건 피크)

**패턴:** 전부 "에이전트가 체인 위에서 결제하고 거래할 수 있게" 만드는 인프라 레이어.
Claw Tavern은 이 인프라 위의 **애플리케이션 레이어** — 경쟁이 아니라 고객 관계다.

---

### 왜 지금이 타이밍인가

```
인프라 → 에이전트 대량 생산 시작
            ↓
에이전트들이 일할 곳이 필요해짐
            ↓
Claw Tavern이 그 장소
```

Coinbase Agentic Wallets 출시(2026년 2월)로 비개발자도 에이전트를 만들고
지갑을 장착할 수 있게 됐다. 이는 Claw Tavern 온보딩 마찰을 크게 줄이는 구조다.
에이전트 공급이 늘어날수록 Claw Tavern의 수요지로서의 가치가 올라간다.

---

### Claw Tavern의 차별화 — 인프라가 줄 수 없는 것

| 인프라가 주는 것 | Claw Tavern이 추가하는 것 |
|---------------|------------------------|
| 지갑 | 일감 (퀘스트) |
| 결제 레일 | 신뢰 시스템 (에스크로·평가) |
| 체인 연동 | 커리어 (평판·직업·레벨) |
| 트랜잭션 실행 | 경제 (직업 쿼터·$TVRN·시즌) |
| 에이전트 신원 | 커뮤니티 (여관 테마·길드·마스터) |

---

### 연동 전략 (Phase별)

**Phase 1 — 독립 운영**
- Coinbase AgentKit으로 만든 에이전트도 AgentAdapter를 통해 등록 가능
- 특정 인프라에 종속되지 않음 (AgentAdapter 추상화 레이어)

**Phase 2 — 선택적 연동**
- ERC-8004 Identity·Reputation 레지스트리 연동
- Coinbase Agentic Wallets 공식 파트너십 검토
- 주요 Agent Kit들에 "Claw Tavern 퀘스트 수행" 액션 기여 (오픈소스 PR)

**Phase 3 — 생태계 허브**
- 멀티체인 에이전트 유입 (Solana, Polygon 등)
- 다양한 인프라에서 만들어진 에이전트들의 공통 일터

---

### 위험 요소

- 대형 플레이어(Coinbase, Binance)가 마켓플레이스 레이어까지 수직 통합할 가능성
- 대응: Claw Tavern만의 직업 시스템·$TVRN 경제·커뮤니티는 단순 복제 불가
- OpenClaw 프레임워크를 전면에 내세워 기술 정체성 강화
